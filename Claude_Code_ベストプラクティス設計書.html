Claude Code ベストプラクティス
基本設計書

Anthropic公式ドキュメント・実運用経験に基づく、Claude Code作業環境の最適な設計指針。
CLAUDE.md・フォルダ構造・コンテキスト管理の社内標準を定める。

SECTION 01

概要・目的

Claude CodeはAnthropicが提供するAIコーディングアシスタントのCLI。ターミナル上でAIと対話しながら、コード作成・ファイル操作・調査・分析などを行える。

この設計書の目的

Claude Codeの性能を最大限に引き出すには、作業環境の設計が極めて重要。特にCLAUDE.mdの書き方とフォルダ構造は、AIの精度・速度・コンテキスト効率に直結する。

本書はAnthropic公式ベストプラクティスおよび社内実運用の知見に基づき、全社員が同じ基準でClaude Codeを活用できるようにするものである。

最重要の制約

Anthropic公式ドキュメントより:

「ほとんどのベストプラクティスは1つの制約に基づく: Claudeのコンテキストウィンドウはすぐに埋まり、埋まるほど性能が劣化する。」

つまり、不要な情報をロードしない設計が全ての基本になる。

SECTION 02

5つの基本原則

1
地図 > 構造
フォルダ構造を複雑にするより、CLAUDE.mdにプロジェクト一覧（地図）を整備する方がAI精度が向上する。35リポジトリ運用の実事例で実証済み。
2
CLAUDE.mdは200行未満
各行に「これを消すとClaudeが間違えるか？」テスト。Noなら削除。超えると指示遵守率が低下する。公式推奨
3
ルールは分離・条件付きロード
.claude/rules/にルールを分離し、paths指定で必要な時だけロード。コンテキスト消費を最小化する。
4
重要ルールにはIMPORTANTを付与
「間違えたら致命的」なルールには**IMPORTANT**マーカーを付けるとClaudeの遵守率が上がる。公式推奨
5
階層は3段まで
4段超でAI・人間ともにナビゲーション効率が崩壊。迷う分類体系は作らず、フラット+地図で管理する。
SECTION 03

フォルダ設計

推奨ディレクトリ構成

プロジェクトルート/
├── CLAUDE.md ← 43行以下。環境・必須ルールのみ
├── CLAUDE.local.md ← 個人設定（.gitignoreに追加）
├── .claude/
│   ├── settings.json ← チーム共有設定
│   ├── settings.local.json ← ローカル設定（.gitignore）
│   ├── rules/ ← パス指定可能なルールファイル群
│   │   ├── project-map.md ← プロジェクト一覧（pathsなし=毎回読込）
│   │   ├── session-behavior.md ← 行動ルール（pathsなし=毎回読込）
│   │   ├── dashboard.md ← ダッシュボード系（paths指定=条件付き）
│   │   └── git.md ← Git系（paths指定=条件付き）
│   └── skills/ ← ワークフロー定義（オンデマンド読込）
│       └── my-skill/
│           └── SKILL.md
├── _archive/ ← 休止プロジェクト（フラット配置）
├── プロジェクトA/ ← ルート直下にフラット配置
├── プロジェクトB/
└── ...

フォルダ設計のポイント

ポイント説明
プロジェクトはルート直下カテゴリフォルダに入れない。フラット配置 + CLAUDE.mdの地図でナビゲーション
_archive/ で休止管理2ヶ月以上更新のないプロジェクトを退避。必要時にルートに戻す
大規模リポは移動しないnode_modules等で数万ファイルあるプロジェクトは、Dropbox同期リスクがあるため固定
散在ファイルは親プロジェクトへルートに作業データを放置しない。該当プロジェクトのフォルダに収納
SECTION 04

CLAUDE.mdの書き方

書くべきもの MUST

内容例
Claudeが推測できないBashコマンドpy（python3は使えない）
環境の癖Pythonパス、OS固有設定、必要な環境変数
Git/PRの慣習ブランチ命名、コミットメッセージ規約、push分割ルール
デプロイ方法GitHub Pages、index.htmlリネームルール
間違えたら致命的なルールファイル削除の扱い、ファクトチェック必須
非自明なカスタム機能スクリーンショット共有（ss機能）

書かないべきもの AVOID

内容理由
Claudeがコードを読めばわかることコンテキストの無駄遣い
標準的な言語規約Claudeは既に知っている
詳細なAPIドキュメントリンクで十分。@pathでインポートも可
頻繁に変わる情報.claude/rules/に分離する
ファイルごとのコードベース説明Claudeが自分で読める
「きれいなコードを書け」等の自明な指示効果がなくノイズになる

書き方のコツ 公式推奨

  • 具体的に書く: 「適切にフォーマット」ではなく「2スペースインデント」
  • マークダウンのヘッダー・箇条書きで構造化する
  • 重要なルールには **IMPORTANT** や **YOU MUST** を付与
  • @path/to/file 構文で外部ファイルをインポート可能（最大5ホップ）
  • HTMLコメント <!-- --> は除去されるため、人間向けメモに使える
  • 定期的にレビューし、矛盾・古い情報を削除する
判断テスト

CLAUDE.mdの各行に対して「これを削除するとClaudeが間違えるか？」と問う。Noなら削除するか、.claude/rules/に移動する。

SECTION 05

.claude/rules/ の活用

3種類の読み込みモード

配置読み込み用途
rules/ pathsなし 毎セッション自動読込 プロジェクト地図、行動ルールなど全般的なもの
rules/ paths指定 該当ファイル操作時のみ 言語別ルール、フレームワーク別ルール
skills/ オンデマンド（呼出時） ドメイン知識、ワークフロー

paths指定の書き方

---
description: ダッシュボード生成プロジェクトのルール
paths:
  - "AIナレッジバンク_サマリー/**"
  - "LIFEFUND_経営分析ダッシュボード/**"
  - "HRbrain_API連携/**"
---
- テンプレートベースでHTMLを生成する（ゼロから生成禁止）
- ライトテーマ（白背景+インディゴアクセント）

このルールは、上記3フォルダ内のファイルを操作する時だけ読み込まれる。それ以外の作業中はコンテキストを消費しない。

パスパターンの例

パターンマッチ対象
**/*.ts全ディレクトリのTypeScriptファイル
src/**/*src/ 以下の全ファイル
*.mdプロジェクトルートのMarkdownファイル
src/**/*.{ts,tsx}ブレース展開で複数拡張子
CLAUDE.md vs rules/ vs skills の使い分け
  • CLAUDE.md: 毎セッション必須。「消すとClaudeが間違える」ルール
  • rules/（pathsなし）: 毎セッション読込。便利だが無くてもエラーにならないもの
  • rules/（paths指定）: 条件付き読込。ファイル種別ごとのルール
  • skills: オンデマンド読込。繰り返し使うワークフロー。500行未満推奨
SECTION 06

コンテキスト管理

Claude Codeの性能はコンテキストウィンドウの使い方で決まる。以下の戦略を守ること。

コンテキスト節約の5つの戦略 公式推奨

#戦略方法
1/clear を頻繁に使う無関係なタスク間でコンテキストをリセット。最も効果的
2サブエージェントで調査委譲調査は別コンテキストで実行し、メインを汚さない
3/compact で圧縮重要情報を指定してカスタム圧縮
42回修正してダメなら/clear失敗アプローチが溜まったコンテキストは性能劣化の原因
5rules/のpaths指定不要なルールをロードしない

よくある失敗パターン

避けるべきパターン
  1. キッチンシンクセッション — 1つのセッションで無関係なタスクを混ぜる → /clearで分離
  2. 繰り返し修正 — 2回修正してダメなら/clearして良いプロンプトで再開
  3. 肥大化したCLAUDE.md — 200行超えるとClaudeが半分を無視 → 容赦なく剪定
  4. 範囲を限定しない調査指示 — 「調査して」はNG。スコープを絞るかサブエージェントで
SECTION 07

運用ルール

新規プロジェクト追加フロー

  1. ルート直下にフォルダを作成する（カテゴリフォルダに入れない）
  2. .claude/rules/project-map.md の地図テーブルに1行追記する
  3. 必要に応じてプロジェクト内に CLAUDE.md を配置（サブディレクトリ用）

プロジェクト休止・完了時

  1. フォルダを _archive/ に移動
  2. .claude/rules/project-map.md から該当行を削除
  3. paths指定のrulesから該当パスを削除

CLAUDE.md読み込みの仕組み

スコープ配置場所読み込み共有範囲
グローバル~/.claude/CLAUDE.md毎回全プロジェクト・自分だけ
プロジェクト./CLAUDE.md毎回チーム（git管理）
ローカル./CLAUDE.local.md毎回自分だけ（.gitignore）
サブディレクトリ./sub/CLAUDE.md遅延（該当ファイル操作時）チーム

サブディレクトリのCLAUDE.mdは遅延読込のため、大規模プロジェクトでもコンテキストを圧迫しない。

定期メンテナンス

頻度作業
毎月CLAUDE.mdの棚卸し（矛盾・古い情報の削除、行数チェック）
毎月_archive/ の見直し（復帰させるもの・完全削除するもの）
四半期rules/ファイルの見直し（pathsの更新、不要ルールの削除）
随時project-map.md の更新（プロジェクト追加・休止時）

テンプレート

CLAUDE.md テンプレート（コピーして使用）

# プロジェクト共通設定

## 言語
- 日本語で対応すること

## 環境
- Python: （パスを記載）
- Node/npm: （バージョン等）
- OS: （OS名とシェル）

## Git / GitHub
- user.name: （ユーザー名）
- user.email: （メールアドレス）
- GitHub: （GitHubユーザー名）

## デプロイ
- （デプロイ方法を記載）

## ファクトチェック必須ルール
**IMPORTANT:** 自分の知識だけで事実情報を断言しない。
必ずWebSearchで裏取りしてから回答する。

## ファイル削除ルール
**IMPORTANT:** Claude Codeにはファイル・フォルダの削除権限がない。
- 「削除」とは、`削除/` フォルダに移動することを意味する

## フォルダ運用
- 新規プロジェクトはルート直下に作成
- プロジェクト一覧は @.claude/rules/project-map.md を参照

rules/ファイル テンプレート（paths指定あり）

---
description: （このルールの1行説明）
paths:
  - "フォルダA/**"
  - "フォルダB/**"
---
- ルール1
- ルール2

rules/ファイル テンプレート（pathsなし=毎回読込）

---
description: （このルールの1行説明）
---
# タイトル

- ルール1
- ルール2
